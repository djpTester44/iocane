# WORKFLOW: FIND DEAD CODE

**Objective:** Systematically scan a Python codebase for orphaned or completely unreferenced implementation files, classes, and configuration. Output a deterministic "Dead Code Audit Report" for review.

**Context:**

* Scope: `src/`, `interfaces/`, and `tests/` directories.
* Architecture: Contract-Driven Development (CDD).
  * `.pyi` files are strictly for static analysis and are never imported at runtime.
  * Ghost Anchors (types defined in `.pyi` or `project-spec.md` with no runtime implementation) are Valid Forward Declarations, not dead code.
  * `TYPE_CHECKING` blocks and their `else` runtime fallbacks (e.g., `Protocol = object`) are structural CDD elements, not dead code.
* Input: `settings.yaml`, `plans/project-spec.md`, and Python source files.

**Procedure:**

1. **TOOL SETUP & EXECUTION:**
   * **Prerequisite Check:** Verify that `scripts/audit_dead_code.py` exists. If it is missing, stop the workflow and inform the user that the CDD auditor script is required.
   * **Execution:** Run the CDD-aware dead code auditor: `uv run python scripts/audit_dead_code.py` (This script requires `vulture` and `pyyaml` in the environment).
   * **Strict Constraint:** Read the markdown report generated by the script. You are strictly forbidden from running raw `vulture` directly, as it circumvents the architecture filters and produces CDD false positives.
   * **Verification:** Conduct rigorous recursive searches across `src/`, `interfaces/`, and `tests/` ONLY to manually verify a specific finding from the script's report. Do not use secondary AST tools.

2. **GHOST ANCHOR VERIFICATION (Manual Fallback):**
   * If the script flags any unmatched `.pyi` definitions, verify them against `plans/component-contracts.toml` (authoritative component list). If the component name appears as a key under `[components]`, it is a Valid Forward Declaration. If absent from the TOML, cross-check `plans/project-spec.md` CRC cards — if still undocumented, flag as an undocumented forward declaration (Tier 2 Defect).

3. **REPORT PRESENTATION:**
   * Present the deterministic Markdown report generated by the script to the user. The report will already be categorized into:
     * **Tier 1: Actionable Dead Code:** 100% confidence dead code safe for deletion (e.g., unreferenced files, dead variables, completely orphaned classes).
     * **Tier 2: Defects & Missing Integrations:** Code that is unwired but should be (e.g., duplicate YAML keys, documented exceptions that are never raised).
     * **Tier 4: Verified Not Dead:** False positives caught during analysis (Ghost Anchors, implicit consumers, structural IO methods).
   * **[CRITICAL RULE] STRICTLY FORBIDDEN AUTOMATIC DELETION**: Do NOT automatically delete any code. The workflow must ONLY output the report for the user to review.
   * End the workflow by instructing the user to manually task out deletions via `/io-plan-batch` based on the accepted findings in the report.
