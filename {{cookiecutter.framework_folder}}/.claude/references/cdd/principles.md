# Contract-Driven Development: Foundational Principles for Agentic Development

## What This Document Is

This document defines Contract-Driven Development (CDD) as the governing
development architecture for this codebase. It is written for any agent
(human or AI) that will read, write, or modify code in this project. If you
are an AI coding agent, these principles override your default behaviors
around interface files, type stubs, and test generation.

---

## The Central Premise

A **contract** is a behavioral specification that exists at a module boundary.
It is the single source of truth for what a module does, what it requires,
what it guarantees, and how it fails. Everything else -- implementation,
tests, documentation -- is derived from or constrained by the contract.

A contract is **not** a type hint file. It is **not** a convenience for the
compiler or type checker. It is an architectural artifact that carries the
same authority as a signed interface agreement between two independent teams.

---

## The Seven Principles

### 1. Contract First, Always

No implementation code is written until a contract exists. No test is written
until a contract exists. The contract is the first artifact produced for any
module, service, or component.

**What this means for the agent:**
- When asked to build a new module, your first output is a contract file
  (`.pyi`, `.contract.ts`, or equivalent), not an implementation.
- When asked to add a feature to an existing module, check whether the
  contract covers it. If not, propose a contract amendment before writing
  implementation code.
- Never infer a contract from existing implementation. The contract is
  prescriptive (it dictates what shall be built), not descriptive (it does
  not document what was already built).

### 2. Contracts Define Behavior, Not Just Shape

A contract specifies five things:

1. **Data types and their invariants** -- the structural shapes that cross
   the boundary, plus the rules that always hold for those shapes (e.g.,
   "email is a valid RFC 5322 address," "created_at <= updated_at").

2. **Function/method signatures** -- the inputs, outputs, and generic
   constraints.

3. **Preconditions (`@pre`)** -- what must be true about inputs before
   invocation. If a precondition is violated, the caller is at fault.

4. **Postconditions (`@post`)** -- what the function guarantees about its
   return value and any side effects. If a postcondition is violated, the
   implementation is at fault.

5. **Error clauses (`@throws` / `@raises`)** -- the specific error types
   that can be raised and the exact conditions that trigger them. Errors
   are part of the contract, not implementation accidents.

**What this means for the agent:**
- When you read a `.pyi` or interface file, do not skip the docstrings.
  The `@pre`, `@post`, `@throws`, and invariant annotations are
  behavioral requirements, not documentation flavor text.
- When you write an implementation, every guard clause, validation check,
  and error raise must trace back to a specific contract clause. If you
  find yourself writing logic that has no contract basis, stop -- the
  contract needs to be extended first.

### 3. Tests Are Derived From Contracts

Tests are the executable verification of a contract. They are generated by
reading the contract and producing a test case for every clause:

- Each `@pre` -> test that violating the precondition is rejected
- Each `@post` -> test that the postcondition holds after a valid call
- Each `@throws` -> test that the error is raised under the stated condition
- Each invariant -> test that the invariant holds across sequences of
  operations, not just single calls
- Boundary/edge cases implied by type constraints (empty strings, maximum
  lengths, zero values, None where not permitted)

**What this means for the agent:**
- When asked to write tests, read the contract file first. Structure the
  test file to mirror the contract's structure -- one test class per method,
  one test per clause.
- Do not inspect the implementation to decide what to test. The contract
  is your only input for test design.
- Every test must carry a comment or docstring citing which contract clause
  it verifies.

### 4. Independent Implementability

Any agent must be able to pick up a contract file and produce a conforming
implementation without reading any other module's internals. If this is not
possible, the contract is underspecified -- fix the contract, not the
implementation.

**What this means for the agent:**
- When implementing a contract, treat it as your complete specification.
  Do not reach into other modules' source files to understand what yours
  should do.
- If the contract references other contracts (e.g., a service contract
  depends on a repository contract), you may read those dependency
  contracts, but not their implementations.

### 5. Substitutability

Any implementation that satisfies the contract can replace any other. This
is the Liskov Substitution Principle applied at the module boundary.

**What this means for the agent:**
- Never allow implementation-specific behavior to leak through the
  contract boundary. If your implementation does something useful that
  the contract doesn't specify, that behavior is incidental and must not
  be relied upon by callers.
- If two implementations of the same contract behave differently in some
  edge case, the contract is ambiguous -- fix the contract.

### 6. Explicit Error Contracts

Failure modes are first-class citizens of the contract. A contract
specifies which errors can be raised and under what conditions.

**What this means for the agent:**
- Never raise an error type that isn't declared in the contract.
- Never swallow an error condition that the contract says must be raised.
- If you discover a failure mode the contract doesn't cover, propose a
  contract amendment.

### 7. Contracts Are Immutable Once Active

Once a contract is agreed upon and implementation/test work begins, the
contract is frozen for that iteration. Changes require explicit
renegotiation -- a proposal, a rationale, and approval before any contract
file is modified.

**What this means for the agent:**
- Never silently widen a contract (add parameters, accept broader inputs).
- Never silently narrow a contract (remove guarantees, restrict outputs).
- If you need to change a contract, produce a separate diff of the
  contract file with a written rationale. Do not bundle contract changes
  with implementation changes.

---

## Workflow Summary

For every new module or feature, the mandatory order is:

```
1. DESIGN    -> Write or locate the contract (.pyi / interface file)
2. VERIFY    -> Review that all five elements are present and sufficient
3. TEST      -> Derive tests from the contract (all tests should fail)
4. IMPLEMENT -> Write code until all contract-derived tests pass
5. AUDIT     -> Confirm every implementation branch traces to a clause
```

Steps cannot be skipped. Steps cannot be reordered.

## Contract File Conventions

| Element              | Where It Lives                                         |
|----------------------|--------------------------------------------------------|
| Module invariants    | Module-level docstring at top of contract file         |
| Data types           | Dataclass / TypedDict / interface definitions          |
| Error types          | Exception subclass declarations with docstrings        |
| Method signatures    | Protocol / ABC / interface method stubs                |
| Pre/postconditions   | `@pre` / `@post` in method docstrings                  |
| Error clauses        | `@throws` / `@raises` in method docstrings             |

---

## What a Contract Is Not

- **Not a type stub for the checker.** Type checkers consume it, but it
  exists for humans and agents, not for mypy.
- **Not generated from implementation.** It is authored deliberately as a
  design artifact. Auto-generating a `.pyi` from source inverts the
  authority relationship.
- **Not optional documentation.** Removing a `@pre` annotation is
  equivalent to removing a requirement -- it changes what implementations
  must do.
- **Not a test plan.** Tests are derived from it, but the contract itself
  is a specification, not a test document.
