# Contract-Driven Development (CDD) Workflow Guide

This workspace is governed by a strict **Macro/Meso/Micro Contract-Driven Development (CDD)** architecture.
The objective is to eliminate "Unanchored Code" (logic without design), prevent context drift, and enforce strict Type-Driven Development (TyDD).

The agent operates as a **Surgical Code Architect**. It does not act autonomously across broad scopes; it requires explicit human verification at critical state-change gates.

## The Macro / Meso / Micro Hierarchy

| Tier | Sub-levels | Artifacts | Persistence |
|------|-----------|-----------|-------------|
| **Macro** (Strategic) | Requirements, Design, Contracts | `PRD.md`, CRC Cards, `.pyi` Protocols | Persistent |
| **Meso** (Session) | Scoped execution boundary | `execution-handoff-bundle.md` | Ephemeral |
| **Micro** (Execution) | Tasks, Implementation | `tasks.json`, `.py` files | Mixed |

**Ownership:**
- **Macro** = user + `/io-architect` (design-time decisions)
- **Meso** = `/io-handoff` (automated scoping with backlog gate)
- **Micro** = `/io-loop` (automated TDD execution)

---

## Directory Architecture

- `plans/PRD.md`: The raw business requirements and origin seed. User-owned.
- `plans/project-spec.md`: **The Living Architecture.** Contains:
  - **Interface Registry:** Maps components to `.pyi` contracts.
  - **Component Specifications:** CRC Cards and Sequence Diagrams (Behavioral Design).
- `plans/PLAN.md`: The high-level strategic roadmap (Checkpoints). No granular tasks.
  - **`- Remediation Backlog`:** Tagged findings from `/review`, consumed by `/io-tasking` before new CPs.
- `plans/tasks.json`: Ephemeral, atomic TDD steps for the current Checkpoint only.
- `plans/progress.md`: Append-only historical log of completed tasks.
- `interfaces/*.pyi`: The binding contracts. Single source of truth for structural types.
- `.agent/`: Contains the rules, templates, skills, and workflows that govern the LLM.
- `plans/execution-handoff-bundle.md`: Runtime artifact generated by `/io-handoff`. Scopes the next implementation session.

## The Workflow Sequence

Execution follows a strict chronological sequence. The **Pre-Entry Gate** (`/review-plan`) is an iterative validation loop run *before* entering the canonical chain. **Do not skip phases within the chain.**

```
[Plan Iteration: write/amend plan -> /review-plan (repeat until PASS)]
    -> /io-architect -> /io-handoff -> /io-tasking -> /io-loop
    -> /review -> /gap-analysis -> /doc-sync
```

### Phase 1: PRD Clarification [Mandatory Gate]

**Command:** `/io-clarify`
**Purpose:** Prevents architectural hallucination by forcing resolution of ambiguous or missing requirements *before* system design begins.

**Agent Actions:**

1. Evaluates `plans/PRD.md` against strict constraints (e.g., success metrics, explicit dependencies, error handling).
2. If ambiguities exist, halts and prompts the user with specific categorical questions.
3. Once all ambiguities are resolved, sets `**Clarified:** True` in the PRD header.

**Human Verification:** Answer the generated questions to unblock initialization or replanning.

---

### Phase 2A: Initialization (Greenfield) [One-Time]

**Command:** `/io-init`
**Purpose:** Transforms a raw `plans/PRD.md` into the architectural foundation for a new project.

> **[HARD GATE]** `plans/PRD.md` must have `**Clarified:** True` in its header. If not, run `/io-clarify` first.

**Agent Actions:**

1. Analyzes the PRD.
2. Generates `plans/project-spec.md` with initial Interface Registry and domain models.
3. Generates `plans/PLAN.md` with high-level Checkpoints (CPs).

**Human Verification:** Review the generated architecture. Modify `project-spec.md` if needed.

---

### Phase 2B: Adoption (Brownfield) [One-Time]

**Command:** `/io-adopt`
**Purpose:** Anchors an existing codebase by reverse-engineering a "Refactor PRD" from the current state.
**Agent Actions:**

1. **Current State Analysis:** Scans code and generates `plans/current-state.md`.
2. **PRD Generation:** Maps the Current State into a Iocane-compliant `plans/PRD.md`.
3. **Handoff:** Stops for user approval.

**Human Verification:** Review `plans/PRD.md`. Once approved, proceed to Phase 2A (`/io-init`).

---

### Phase 3: Pre-Entry Gate: Plan Review [Iterative]

**Command:** `/review-plan`
**Purpose:** Self-healing validation that an implementation plan maintains CRC-Protocol symmetry and follows CDD principles. Auto-remediates design-level violations; escalates architectural violations to the user.

**Trigger:** Run whenever an implementation plan exists and needs validation -- gap analysis remediation, post-code-review fixes, ad-hoc planning, or new checkpoint preparation.

**Self-Healing Flow:**

1. Write or amend the implementation plan.
2. Run `/review-plan` -- runs 7 checks and classifies violations as auto-remediable or not.
3. Auto-remediable violations (e.g., missing CRC anchor, orphaned design) are fixed inline and re-validated (max 3 iterations).
4. Non-auto-remediable violations (e.g., layer violations, DI wiring) escalate to the user.
5. On PASS: proceed to `/io-architect`.

**Agent Actions:**

1. Loads CRC Cards, Protocols, Interface Registry, and `pyproject.toml [tool.importlinter]` layer contracts.
2. Runs 7 checks: Private Method Gate, CRC-Protocol Symmetry, Atomicity, Layer Boundary Compliance, DI Compliance Preview, False Positive Justification.
3. Auto-remediates fixable violations, logs changes with `[AUTO-AMENDED]` markers.
4. Outputs a PASS/FAIL report. On PASS, stamps plan with `**Plan Validated:** PASS`.

**Gate Behavior:** Auto-remediable VIOLATIONs are fixed inline. Non-auto-remediable VIOLATIONs block entry to the canonical chain and require user intervention.

**Human Verification:** Review the Self-Healing Log in the plan to confirm auto-amendments are correct.

---

### Phase 4: Architecture & Design [Iterative]

**Command:** `/io-architect`
**Purpose:** Anchors **Behavior** (Design) before **Structure** (Contract).

**Iteration Triggers:**

- New Checkpoint from `plans/PLAN.md`.
- `[DESIGN]` or `[REFACTOR]` items in the Remediation Backlog (see [Remediation Backlog Lifecycle](#remediation-backlog-lifecycle)).
- Drift correction when implementation reveals a flawed Spec.

**Agent Actions:**

1. Checks `plans/PLAN.md - Remediation Backlog` for pending `[DESIGN]`/`[REFACTOR]` items and includes them in context.
2. Generates a **CRC Card** and **Critical Sequence Diagram**, appended to `plans/project-spec.md`.
3. Translates design into a rigid `.pyi` Protocol in `interfaces/`.
4. Outputs a soft reminder to check the backlog before running `/io-tasking`.

**Human Verification:** Run `uv run mypy interfaces/` to verify type contracts.

---

### Phase 5: Execution Handoff [Iterative]

**Command:** `/io-handoff`
**Purpose:** Slices the macro architecture into a strictly bounded meso-state for the next implementation session.

> **[HARD GATE]** Checks `plans/PLAN.md - Remediation Backlog` before proceeding:
>
> - `[DESIGN]` or `[REFACTOR]` items --> **HALT.** Run `/io-architect` first.
> - `[CLEANUP]` items --> Set active path to "Backlog Cleanup".
> - Empty backlog --> Proceed to next sequential Checkpoint.

**Agent Actions:**

1. Identifies the active scope (backlog cleanup or next incomplete Checkpoint).
2. Defines session boundaries: target implementation path, read-only `.pyi` interfaces, restricted zones.
3. Generates `plans/execution-handoff-bundle.md` with the scoped session goal, allowed files, and preload targets.
4. Outputs instruction to run `/io-tasking` next.

**Human Verification:** Review the handoff bundle to confirm scope boundaries are correct.

---

### Phase 6: Task Generation [Iterative]

**Command:** `/io-tasking`
**Purpose:** Generates atomic TDD tasks strictly bounded by the handoff bundle.

> **[CONSTRAINT]** `/io-tasking` is **blinded** to `plans/PLAN.md` and `plans/project-spec.md`. It reads ONLY from `plans/execution-handoff-bundle.md`. The backlog gate is enforced in Phase 5 (`/io-handoff`), not here.

**Agent Actions:**

1. Reads `plans/execution-handoff-bundle.md` to ingest the scoped session goal and allowed files.
2. Maps the goal into the 4-step TDD cycle: `setup`, `test`, `implement`, `refactor`.
3. Overwrites `plans/tasks.json` with atomic implementation steps.
4. Locks `context_files` per task to a strict subset of the handoff bundle's Preload Targets.

**Human Verification:** Review `tasks.json` to ensure steps cover all session goal responsibilities.

---

### Phase 7: Execution Loop [Iterative]

**Command:** `/io-loop`
**Purpose:** Executes the current atomic task using strict TDD, halting only for cascading failures or architectural drift.

**Task Types:**

| Type | Goal | Gate |
|---|---|---|
| `setup` | Create skeleton files, interfaces, or configs | `uv run mypy` clean |
| `test` | Write a failing test (Red state) | `uv run pytest` MUST fail |
| `implement` | Write minimum code to pass the test (Green state) | `uv run pytest` + `uv run mypy` pass |
| `refactor` | Clean up for SOLID/DI compliance | `ruff`, `mypy`, `lint-imports`, `check_di_compliance` clean |
| `verify` | Run a command and confirm expected output (e.g., E2E) | Command succeeds |

**Agent Actions:**

1. Reads the first pending task in `plans/tasks.json`.
2. Executes the task according to its type and gate (see table above).
3. On implementation failures, attempts up to 3 autonomous remediations before escalating.
4. Updates `tasks.json` and logs to `plans/progress.md` via `.agent/scripts/log_progress.sh`.
5. Automatically loops to the next pending task.

**Drift Escalation:** If an untracked `# noqa: DI` escape hatch is detected, `/io-loop` **halts immediately** and requires the user to add a `[REFACTOR]` or `[CLEANUP]` ticket to the Remediation Backlog before execution can resume.

**Human Verification:**

- For Tests: `uv run pytest <test_file>` -- verify failure.
- For Implementation: `uv run pytest` -- verify success.

### Phase 8: Code Review [Iterative]

**Command:** `/review` > `/review-capture`

**`/review` Purpose:** Systematic code review verifying code matches the Structural Contract (`.pyi`) and Behavioral Design (CRC).

**Agent Actions:**

1. Loads Protocol (`.pyi`) and CRC from `project-spec.md` as anchors.
2. Runs pre-scans: `extract_structure.py`, `lint-imports`, `check_di_compliance.py`.
3. Analyzes behavior vs. CRC responsibilities, correctness, types, style, SOLID/DRY, security, and tests.
4. Outputs a Findings table (Severity / Location / Issue / Recommendation).
5. **Mandatory:** Runs `/review-capture` to log all findings to `PLAN.md`.

**`/review-capture` Purpose:** Classifies findings and appends them to `plans/PLAN.md - Remediation Backlog`.

**Classification tags:**

Findings are classified according to the definitions and routing logic in `.agent/rules/ticket-taxonomy.md`.

**Human Verification:** Review the findings table. Decide which items to act on. The backlog is the formal tracking record -- no action is lost.

---

### Phase 9: Gap Analysis [Iterative]

**Command:** `/gap-analysis`
**Purpose:** Verifies implementation fidelity by identifying gaps between requirements, design, contracts, and code.

**Agent Actions:**

1. Audits Code vs. Design vs. Contracts using `extract_structure.py`.
2. Runs `check_di_compliance.py` to flag DI violations.
3. Outputs a Gap Analysis Report categorized by: Structural Gaps, Behavioral Gaps, Compliance Gaps.
4. Asks the user which gaps to act on, then routes selected items to `/review-capture` for backlog entry.

**Human Verification:** Review the report. Select items for backlog capture. Run `/io-handoff` to begin remediation.

---

### Phase 10: Documentation Sync [Iterative]

**Command:** `/doc-sync`
**Purpose:** Reconciles all documentation with current codebase state after a Iocane Loop phase completes.

**Agent Actions:**

1. Updates `README.md` (identity only, follows template).
2. Checks `project-spec.md` Interface Registry and CRC Cards for gaps.
3. Runs `check_design_anchors.py` to verify behavioral anchors match structural contracts.
4. **Backlog Reconciliation (auto):** For each `[ ]` item in `PLAN.md - Remediation Backlog`, verifies whether the fix is present in the codebase. Marks confirmed fixes `[x]` in-place -- no approval needed.
5. Proposes checkpoint/roadmap updates to `PLAN.md` (user approval required).
6. Verifies file links in markdown docs.

**Human Verification:** Approve any proposed `PLAN.md` checkpoint/roadmap changes.

---

### Out-of-Band: Replanning [On-Demand]

**Command:** `/io-replan`
**Purpose:** Propagates PRD changes to `PLAN.md` and `project-spec.md` without destroying existing work. This is **not** part of the linear execution flow -- it is triggered only when the PRD itself changes.

> **[HARD GATE]** The updated `plans/PRD.md` must have `**Clarified:** True` in its header. If the update introduces new ambiguities, run `/io-clarify` first.

**Agent Actions:**

1. Generates a "Change Report" (New/Modified/Removed features) based on the PRD delta.
2. Proposes changes to `plans/PLAN.md` (requires user approval).
3. Updates Interface Registry, Mermaid Graphs, and CRC in `plans/project-spec.md`.
4. Prompts to run `/io-architect` next.

**Human Verification:** Review the Change Report and approve edits to `plans/PLAN.md`.

## Remediation Backlog Lifecycle

The `plans/PLAN.md - Remediation Backlog` is the formal tracking record for all `/review` findings. It lives in `## 3. Remediation Backlog` of `PLAN.md`, above all checkpoint phases, so every planning agent reads it first.

```
/review          --> surfaces findings
/review-capture  --> appends [ ] items to PLAN.md backlog with [DESIGN]/[REFACTOR]/[CLEANUP]/[DEFERRED] tags
/io-handoff      --> hard gate: routes by tag before any new CP work
/io-tasking      --> generates tasks from the scoped handoff bundle
/io-loop         --> executes the fix
/doc-sync        --> marks fixed items [x] in-place (auto, no approval)
```

**Rules:**

- Items are **never deleted or moved** the backlog is a permanent audit trail.
- `[x]` items = history. `[ ]` items = active work queue.
- A group (`#### From <CP> /review`) is considered resolved when all its items are `[x]`.

## Critical Guardrails

1. **Stop-Gates Are Mandatory:** The agent must stop after generating a design, contract, plan, or code block. No continuous autonomous loops.
2. **No Implementation Without Design:** Business logic requires a CRC/Sequence in `project-spec.md` AND a Protocol in `interfaces/`.
3. **Context Hygiene:** During `/io-loop`, the agent reads only files listed in the task's `context_files` array.
4. **Backlog Gate:** `/io-handoff` will not proceed to a new CP while `[DESIGN]` or `[REFACTOR]` backlog items remain unresolved.
